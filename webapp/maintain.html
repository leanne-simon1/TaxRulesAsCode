Ways to view and change the rules

Each approach below trials one option from the project plan. All three read from the same live engine, so none can drift from the rules that actually run — and the ones that change anything produce a plain diff a specialist can review, not a code commit they have to write.

1 · Ruleset & parameters editor

The markdown-contract approach. Reads each rule's plain-language statement and its dated policy values (time limits, thresholds, rates). Change a value in a form and it generates the exact YAML edit ready to hand to the maintainer, with a link to the current file on this server — CI tests still gate it. This is the lowest-tooling, highest-leverage option.

Open the editor

2 · Decision-flow diagrams

The auto-generated diagram approach. Builds a Mermaid flowchart of how a result is computed — every rule, input and policy parameter it depends on — straight from the engine's own trace. Because it's generated, the picture can never fall out of step with the code.

Open the diagrams

3 · Legislation Explorer

The browsable reference approach, modelled on OpenFisca's Legislation Explorer. Search every variable and parameter, read its documentation and legislative references, and click through the dependencies from one rule to the next.

Open the explorer

The maintenance loop

  1. A specialist reads the rule in plain language (explorer) and sees how it's computed (diagram).
  2. To change a legislated value, they use the editor to produce the YAML diff.
  3. The generated YAML goes to whoever maintains the repository; the engine's YAML tests in CI gate the change before it can affect any answer.
  4. Every view here refreshes from the live engine, so the picture updates the moment the change lands.
Proof of concept only. Outputs are guidance to explore Rules as Code, not tax advice.
