The rules engine API

Every tool on this site is a thin client of one governed rules engine. The tax rules live once — encoded from plain-language rulesets in OpenFisca, with a legislative reference on every rule — and are served over a small HTTP API. This page is the developer reference: the base URL, the endpoints, the shape of a request, worked examples in three languages, and a console to call the live engine yourself. To browse the rules and values themselves, see the rules model.

What the engine holds

…

Variables. Every fact and every rule is a named variable. Some are inputs; the rest are rules — formulas that compute one thing from others. Browse them in the rules model.

…

Parameters. Policy values that change over time (rates, thresholds, objection time limits) are dated YAML values, each with its own value-over-time history.

…

Entities. Variables are computed for an entity (here, a single generic tax entity) over a period — one case or thousands in a single vectorised call.

Traceable to law

References. Every variable and parameter carries a citation, so answers arrive with their sources — what lets an AI assistant ground its replies.

Base URL & access

The engine is reached under a relative api/ path, so the tools work unchanged behind a reverse proxy. From a browser on this site, that is all you need: 

api/

There is no authentication and no rate limit on this proof-of-concept deployment — it serves only public rules metadata and stateless calculations, holding no personal data. To run it yourself, the whole engine is one container: docker compose -f docker-compose.home.yml up (see DEPLOY.md), which serves this same API on localhost:8850/api/. Because it is one origin, no CORS configuration is needed.

Anatomy of a request

A POST api/calculate body is a situation: one or more entities, each carrying the variables you know as { period: value }, and the outputs you want set to null — the engine fills those in. Periods are a year ("2024") or a month ("2024-02") depending on the variable's definition. 

{
  "tax_entities": {              // a name you choose for this case
    "my_case": {
      "ato_decision_type":     { "2024": "income_tax_assessment" },  // input
      "decision_notice_date":  { "2024": "2024-02-15" },            // input
      "taxpayer_kind":         { "2024": "individual" },            // input
      "has_objection_right":   { "2024": null },   // output — please compute
      "objection_deadline":    { "2024": null },   // output — please compute
      "objection_in_time":     { "2024": null }    // output — please compute
    }
  }
}

The response echoes the same shape with the nulls replaced by computed values. POST api/trace takes an identical body and additionally returns the dependency tree behind each result. Which variables are inputs vs outputs, and what periods they use, is all discoverable from GET api/variables and the rules model.

Response

Response status codes

CodeMeaning
200Success — the body holds the computed situation (or the requested metadata).
400Malformed JSON, or a situation the engine can't parse.
404Unknown variable or parameter name in the path.
422Valid JSON but an invalid situation — e.g. an unknown enum value, a bad date, or a value at the wrong period.
500The engine hit an error while computing. Shouldn't happen for well-formed input; worth reporting if it does.

Error bodies are JSON describing the problem; the tools here surface that text directly.

How the tools here use it

  • Calculators (PAYGI, objection, CGT) collect inputs, call POST api/calculate, and show the answer with the reference URLs of the variables involved as “why” links.
  • Show the working (inside the PAYGI calculator) calls POST api/trace to render the full dependency tree — the transparency layer a chatbot cites to say why, not just what.
  • The objection calculator's eligibility help batches one calculate across every decision type to generate its catalogue live — nothing in it is hand-written tax content.
  • The rules model and maintenance tools read variables and parameters metadata so specialists can review the rules without reading Python.

Next phase: expose these same endpoints to an LLM as tools (direct HTTP tool-use or a small MCP wrapper) so a chatbot converses to collect inputs, calls the engine for the decision, and answers with the engine's references as citations — never computing eligibility itself.
