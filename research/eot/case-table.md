# s 14ZX extension-of-time — case authority base

Compiled 2026-07-15. A **closed, traceable authority list** for the EOT ruleset
and any AI assistant grounded on it (the "case table" pattern from the internal
EOT Copilot pack, rebuilt here from publicly verifiable sources with URLs).
The cases inform the ruleset's *patterns* (`Extension of Time Ruleset.md`
§1.7); none of them, and no combination of them, decides an individual request.

**Verification method.** ato.gov.au, austlii.edu.au and jade.io all block
non-browser fetchers from this environment (HTTP 403 / Cloudflare challenge),
so each row below was verified through multiple independent search-result
sources on 2026-07-15; the URLs follow each provider's stable citation scheme
and should be click-checked once in a browser before formal reliance. Rows
marked ⚠ carry an open verification item.

| *Hunter Valley Developments Pty Ltd v Cohen* [1984] FCA 176; (1984) 3 FCR 344 | FCA, Wilcox J | The aggregated extension-of-time principles (developed under ADJR s 11, applied across statutory EOT discretions): extension is not automatic — an **acceptable explanation** making it fair and equitable is a precondition; a person who **kept the decision maker aware they contested finality** is distinguished from one who let the matter appear concluded (the "resting on rights" pattern); prejudice to the respondent, the merits, and fairness to others in like positions all weigh | Principles case | [AustLII](http://classic.austlii.edu.au/au/cases/cth/FCA/1984/176.html) · [Jade summary](https://jade.io/summary/mnc/1984/FCA/186) ⚠ (Jade shows mnc [1984] FCA 186 vs FCA 176 elsewhere — retrospective numbering differs by provider; the FCR cite (1984) 3 FCR 344 is the stable one) · [Wikipedia overview](https://en.wikipedia.org/wiki/Hunter_Valley_Developments_Pty_Ltd_v_Cohen) |

## Can this case data sense-check s 14ZX decisions?

Yes — in three governed ways, all implemented in the ruleset, and one way it
must never be used:

1. **As the pattern source for the flags.** The atypical-refusal /
   atypical-grant flags encode the *direction* repeatedly visible in these
   cases (Brown's balancing; Windshuttle's low arguable-case bar; Hunter
   Valley's resting-on-rights distinction). The cases justify the flags; the
   flags cite the cases (see the flag variables' `documentation` served by
   `/variable/<name>`).
2. **As a closed citation library for AI.** A Copilot grounded on the engine
   quotes authorities from this table (engine metadata + ruleset §1.7) instead
   of free-generating case law — the "closed, traceable authority base" goal.
3. **As regression tests.** Decided cases can be re-expressed as YAML test
   rows (inputs = the case's factor profile; outputs = which flags fire, or
   none). *Primary Health Care* is the useful discipline case: a large delay
   that was nonetheless extended — it must NOT trip the atypical-grant flag,
   because the explanation factors were found in the taxpayer's favour
   (encoded as `tests/eot/sense_check.yaml`'s "ATO conduct" and mixed-profile
   scenarios; a dedicated PHC-profile row is a good next addition).
4. **Never as a predictor.** Mapping a new request onto "most similar past
   case → same outcome" would be scoring by another name and would fetter the
   discretion (ruleset rule 1.4.1; the internal pack's own template-lock).
   The case data describes how factors have been *weighed*; it cannot supply
   the weighing for a new case.
